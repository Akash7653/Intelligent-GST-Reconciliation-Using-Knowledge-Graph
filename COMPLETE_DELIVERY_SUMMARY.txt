## ðŸ“Š Data Model

### Neo4j Graph Schema
```
Node: Taxpayer
â”œâ”€â”€ name (String)
â”œâ”€â”€ riskScore (Integer)
â””â”€â”€ riskLevel (String: HIGH/MEDIUM/LOW)

Node: Invoice
â”œâ”€â”€ invoice_number (String)
â”œâ”€â”€ gst_amount (Float)
â””â”€â”€ supplier (String)

Node: GSTR3B
â”œâ”€â”€ period (String: Q1-2024, etc.)
â””â”€â”€ filed_date (Date)

Relationships:
- Taxpayer -[:RECEIVED]-> Invoice
- Invoice -[:REPORTED_IN]-> GSTR3B
- Taxpayer -[:ISSUED]-> Invoice
```

### Cypher Query Patterns
```
1. Profile:        MATCH (t:Taxpayer {name: $name}) RETURN t.*
2. Unreported:     MATCH (t:Taxpayer)-[:RECEIVED]->(i) 
                   WHERE NOT (i)<-[:REPORTED_IN]-()
                   RETURN COUNT(i)
3. Circular:       MATCH (t:Taxpayer)-[:RECEIVED]->(i)-[:ISSUED]->(s:Taxpayer)
                   WHERE s.name <> t.name
                   RETURN COUNT(DISTINCT s.name)
4. Reported:       MATCH (:Taxpayer {name})-[:RECEIVED]->(:Invoice)-[:REPORTED_IN]->(:GSTR3B)
5. Unreported:     MATCH (:Taxpayer {name})-[:RECEIVED]->(:Invoice)
                   WHERE NOT (i)<-[:REPORTED_IN]-()
```

## ðŸŽ“ Key Technical Decisions

### 1. Multi-Hop Queries
**Why**: Reveals fraud patterns invisible in traditional databases
- 2-hop shows unreported invoices
- 3-hop reveals circular trading
- Pattern recognition at relationship level

### 2. Explainable AI
**Why**: Auditors need justification for risk flags
- Human-readable summaries
- Categorized reasons
- Compliance documentation

### 3. Modal-Based UI
**Why**: Better UX than separate pages
- Context preserved
- Faster interaction
- Professional feel

### 4. Parallel API Queries
**Why**: Minimizes latency
- 5 Neo4j queries run simultaneously
- Combined response time ~600ms
- While sequential would be ~1200ms

### 5. Color-Coded Risk
**Why**: Visual risk assessment
- RED = HIGH risk
- YELLOW = MEDIUM risk
- GREEN = LOW risk
- Instant understanding
